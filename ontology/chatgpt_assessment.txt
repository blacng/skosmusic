Key Modeling Issues

1. Mixing SKOS Concepts and OWL Classes

Example:

mv:Venue a owl:Class , skos:Concept .

This is OWL punning, but here it is being used to mix:
	•	concept taxonomy
	•	instance typing

This causes problems for reasoning and querying.

Why this matters

You currently have two parallel structures:

VenueType (SKOS)
Venue (OWL class)

But then:

mv:hasVenueType → range mv:VenueType

Yet instances use:

mv:hasVenueType mv:Arena

But Arena is an OWL class, not a SKOS concept instance.

So the model is inconsistent.

Minimal fix

Define VenueType as a class, not a concept.

Then the hierarchy becomes clean:

VenueType
  ├ IndoorVenue
  │   ├ Arena
  │   ├ ConcertHall

SKOS can still provide labels.

⸻

2. Amenity Modeling Is Incomplete

You defined:

mv:Amenity (class)

But you use:

mv:Bar
mv:Restaurant
mv:Elevator
mv:WheelchairAccessible

These are never declared.

Currently they are implicit individuals, which is ambiguous.

Minimal fix

Declare them as instances of Amenity.

⸻

3. Missing Event Model

Your ontology describes venues but not the events occurring in them.

Without this, you cannot answer core competency questions like:
	•	Which venues hosted a given artist?
	•	What concerts occurred at a venue?
	•	What is the upcoming event schedule?

This is a major missing capability.

Minimal addition

Introduce:

MusicEvent
performedAt
eventDate

This is core to the domain.

⸻

4. Geographic Modeling

You created your own location model:

Location
Country
City
Continent

This is fine but incomplete.

Two improvements:

Problem A

No link from Venue → City → Country → Continent path reasoning.

Problem B

No coordinates.

Without coordinates you cannot answer:
	•	nearest venues
	•	distance queries
	•	geographic search

Minimal solution

Add:

geo:lat
geo:long

or reuse schema:GeoCoordinates.

⸻

5. Venue Type Property Redundant

You currently have:

mv:hasVenueType

But the venue type is already implied by the class.

Example:

mv:MadisonSquareGarden a mv:Arena

Then hasVenueType is unnecessary.

Better modeling:

mv:MadisonSquareGarden a mv:Arena

instead of

mv:MadisonSquareGarden a mv:Venue
mv:hasVenueType mv:Arena


⸻

6. Facility vs Venue Hierarchy

Currently:

Facility
 └ Amenity
 └ Venue

This is conceptually odd.

Facilities typically belong to venues rather than being parallel entities.

Better model:

Place
  └ Venue
Amenity

But this change is not strictly required.

⸻

7. Missing Inverse Relationships

You defined:

offersAmenity
isLocatedIn

But no inverses.

Inverse properties improve querying.

Example:

amenityOf
venueLocatedIn

Ontology Patch (Turtle)

### New Class

mv:MusicEvent
    rdf:type owl:Class ;
    rdfs:label "Music Event" ;
    rdfs:comment "A live music performance occurring at a venue." .

### New Properties

mv:performedAt
    rdf:type owl:ObjectProperty ;
    rdfs:domain mv:MusicEvent ;
    rdfs:range mv:Venue ;
    rdfs:label "performed at" .

mv:eventDate
    rdf:type owl:DatatypeProperty ;
    rdfs:domain mv:MusicEvent ;
    rdfs:range xsd:dateTime ;
    rdfs:label "event date" .

### Inverse Properties

mv:amenityOf
    rdf:type owl:ObjectProperty ;
    owl:inverseOf mv:offersAmenity ;
    rdfs:domain mv:Amenity ;
    rdfs:range mv:Venue ;
    rdfs:label "amenity of" .

mv:venueLocatedIn
    rdf:type owl:ObjectProperty ;
    owl:inverseOf mv:isLocatedIn ;
    rdfs:domain mv:Location ;
    rdfs:range mv:Venue ;
    rdfs:label "venue located in" .

### Explicit Amenity Instances

mv:Bar rdf:type mv:EatingOrDrinkingEstablishment .
mv:Restaurant rdf:type mv:EatingOrDrinkingEstablishment .
mv:ConcessionStand rdf:type mv:EatingOrDrinkingEstablishment .
mv:Elevator rdf:type mv:AccessibilityFeature .
mv:WheelchairAccessible rdf:type mv:AccessibilityFeature .
mv:AssistiveListeningDevice rdf:type mv:AccessibilityFeature .
mv:GreenRoom rdf:type mv:Room .
